SYSTEM OVERVIEW
================================================================================
This project is a production-oriented chest X-ray analysis system with a clean
split between fast prediction, explainability, reporting, and evaluation.

Current design:
- ONNX model for fast multi-label disease prediction
- PyTorch `.pth` model for Grad-CAM explainability
- Dockerized Linux backend for the CTransCNN / MMCV runtime
- FastAPI backend with async REST endpoints
- Next.js frontend with disease selection and heatmap views
- RAG-based report generation with OpenAI
- LaTeX PDF report generation
- Bounding-box comparison tooling for Grad-CAM sanity checks

--------------------------------------------------------------------------------
DATA FLOW
--------------------------------------------------------------------------------
1. User uploads a chest X-ray through the frontend.
2. Frontend calls `POST /api/upload`.
3. Backend stores the image and creates a session.
4. Frontend calls `POST /api/inference`.
5. Backend runs ONNX inference with `models/ctranscnn_1.onnx`.
6. Frontend displays disease probabilities.
7. User selects a disease and clicks Explain.
8. Backend tries the PyTorch Grad-CAM path using `CTransCNN/models/epoch_45.pth`.
9. If PyTorch explainability is unavailable, backend falls back to ONNX occlusion.
10. Frontend shows original / heatmap / overlay views.
11. User can generate a medical report.
12. Backend uses RAG, then renders a LaTeX PDF.
13. Frontend allows PDF download.

--------------------------------------------------------------------------------
EXPLAINABILITY FLOW
--------------------------------------------------------------------------------
The explainability path is intentionally split:

- Prediction path:
  - ONNX only
  - fast, simple deployment

- Explainability path:
  - `.pth` model for Grad-CAM
  - loaded only when the user requests explanation
  - cached per session / disease selection

Fallback behavior:
- If PyTorch Grad-CAM cannot load, the backend uses ONNX occlusion.
- This preserves usability even when the CTransCNN runtime is unavailable.

Evaluation support:
- `scripts/compare_gradcam_bbox.py` compares Grad-CAM output against NIH bbox
  annotations from `ref/ref.csv`.
- The script saves:
  - original image + bbox
  - Grad-CAM heatmap
  - Grad-CAM overlay
  - JSON metadata with a rough bbox-energy score

RUNTIME / DEPLOYMENT
--------------------------------------------------------------------------------
Primary deployment pattern:
- Frontend runs on macOS or any local machine
- Backend runs in Docker on Linux-compatible runtime
- Docker backend is used for CTransCNN / MMCV / PyTorch Grad-CAM

Useful runtime notes:
- `CTRANS_PYTHON` can point to an external CTransCNN Python interpreter
- `LATEX_BIN` can override the PDF compiler path if needed

--------------------------------------------------------------------------------
DESIGN PRINCIPLES
--------------------------------------------------------------------------------
- Separation of concerns: prediction, explainability, reporting, evaluation
- Modular: each function is isolated and easy to swap
- Fast prediction path: ONNX stays the default inference engine
- Better explainability: `.pth` Grad-CAM is used only when requested
- Resilient fallback: ONNX occlusion keeps the app usable if PyTorch fails
- Professional output: LaTeX PDF reports and polished UI
- Evaluation-friendly: bbox comparison tooling for visual sanity checks
